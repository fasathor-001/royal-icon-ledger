Admin Dashboard — Onboarding Monitor
Soon

A lightweight internal dashboard to monitor the onboarding funnel without needing to open Supabase manually. Critical before any public posting where signups could spike.

→ View early_access_leads — name, email, country, income_type, status
→ View invite_codes — which codes are used/unused, tied to which emails
→ View user_data — who has completed onboarding (has currency + incomeType set)
→ Drop-off indicator — invited but never activated
→ One-click approve/invite from waitlist
No SQL changes needed. All data already exists — early_access_leads, invite_codes, and user_data are live tables. The dashboard is a read-only front-end over existing data.
Files to create: src/admin/AdminDashboard.jsx src/admin/AdminRoute.jsx
Protected by: service-role key or owner auth check
Currency Expansion — Data-Driven
Soon

Expand from 10 to ~25–30 currencies based on what your first users' country data shows. Do NOT expand blindly — let the data tell you which currencies to add.

→ Step 1: Check early_access_leads.country field after first wave
→ Step 2: Add only the currencies that appear in that data
→ Step 3: Add corresponding timezone entries to match
→ Step 4: Redesign currency tiles to be compact (smaller grid)
No SQL changes needed. Currency is stored as a plain string inside the user_data JSONB column. The database doesn't validate it. Adding INR to currency.js means the app stores "INR" automatically — no schema changes.
Files affected: src/lib/currency.js src/lib/timezones.js src/components/Onboarding.jsx
Candidate currencies: INR, BRL, MXN, PHP, IDR, MYR, PKR, KES, GHS, TRY, SEK, NOK, CHF, NZD, HKD, EGP, ZMW

Product Roadmap — Longer Term
Closed beta → Public launch
Pro Tier — Subscription Pricing
Later

Pricing research to be conducted during closed beta. Primary revenue tier. Do not set pricing before you have real usage data — let users tell you what they'd pay.

Core — Free
Local device. All core modules. Offline-first. No sync. User acquisition at zero cost.
Pro — Subscription
Cloud sync, multi-device, push notifications, sessions. Primary revenue tier.
Business tier (future): multi-user household, shared envelopes, accountant access, API
Bank Feed Integration
Later

Currently mentioned in the FAQ as "on the roadmap." Manual income logging is intentional for the beta — it improves financial awareness. Automate only after the manual system is validated.

Research: Plaid (US/UK), Mono (Nigeria), Stitch (SA), TrueLayer (EU)
Risk: automation removes the intentionality that makes the system work
Approach: offer as opt-in, not default

Bundle Size Warning
Low priority

Every build shows: "Some chunks are larger than 500 kB after minification." The main bundle is ~1.2MB (332KB gzipped). This is acceptable for a PWA but should be addressed before public launch.

Solution: Code-split with dynamic import() — lazy-load tabs that aren't immediately visible
File: src/App_v2.jsx — split Budget, Goals, Trading, MonthlyReview into lazy chunks
Impact: Faster initial load, especially on mobile networks
